# ミート&イート プロダクト概要書

最終更新: 2025-12-20

## 1. プロダクト概要

「ミート&イート」は、日程調整（調整さんのような体験）から集合場所の最適化、近辺の飲食店提案までを、**ログイン不要**で一気通貫に行える日程調整プロダクトです。

ユーザーはカレンダー型 UI で直感的に参加可能日を入力できます。入力時に、居住エリアなどの情報を**ブラウザデータから自動取得**（もしくは手入力）し、その情報をもとにシステムが**最適な集合場所**を導出します。さらに、集合場所周辺の飲食店を**2〜3 件レコメンド**し、食べたくないもの・好みの入力も踏まえて**AI が最適なお店**を提案します。

## 2. 解決したい課題

- 日程が決まっても「どこで会うか」で再調整が発生しがち
- 参加者の居住地が分散していると集合場所の合意形成が難しい
- お店選びは好み・苦手・予算・アクセスなど制約が多く、提案が属人化しやすい

## 3. 想定ユーザー

- 友人同士の食事会/飲み会の幹事・参加者
- 複数人で集まるコミュニティ/サークル
- 社内の数懇親（※ログイン不要運用を前提）

## 4. 価値提案（提供する体験）

- **日程入力が早い**: カレンダー型 UI で直感的に可否を入力
- **場所が決まる**: 参加者の居住エリア情報から集合場所を自動最適化
- **店が決まる**: 周辺から 2〜3 件を提案し、好み/NG を踏まえ AI が最適案を提示
- **始めやすい**: ログイン不要で、URL 共有だけで参加できる

## 5. 主要なユーザーフロー

### 5.1 幹事（作成者）

1. 調整を作成（タイトル/候補日程など）
2. 共有 URL を発行し、参加者へ送付
3. 回答が集まる
4. システムが集合場所を導出し、飲食店候補を提示
5. 幹事が候補から選択（または AI 提案を採用）し確定

### 5.2 参加者

1. 共有 URL を開く（ログイン不要）
2. 居住エリアを自動取得（許可しない場合は手入力）
3. 食の好み/NG（例: 苦手食材、ジャンル、アレルギー等）を入力（任意）
4. カレンダーで参加可能日程を入力
5. 送信

## 6. 機能要件（MVP）

### 6.1 日程調整

- カレンダー型 UI で可否入力
- 回答の集計表示（誰がどこに入れたか/どの日が強いか）

### 6.2 居住エリア情報の取得

- ブラウザからの自動取得（例: 位置情報の許可を求める）
- 自動取得できない/許可されない場合は手入力で代替

### 6.3 集合場所の最適化

- 参加者の居住エリア情報をもとに、集合場所を 1 点導出
- 出力は「最寄り駅/エリア名」などユーザーが理解しやすい形式を優先

※最適化の定義はシンプルに開始します（例: 距離/所要時間の総和が小さくなる地点）。

### 6.4 飲食店レコメンド（2〜3 件）

- 集合場所周辺の飲食店を 2〜3 件提示
- 参加者が入力した好み/NG を反映
- AI が「なぜこの店が良いか」を短く説明して提案（任意テキスト）

### 6.5 ログイン不要

- アカウント作成・ログインなしで作成/回答が可能
- 共有 URL で同一調整にアクセス

## 7. 画面/情報設計（概要）

- LP（説明・開始導線）
- 調整作成画面
- 回答画面（カレンダー入力 + 居住エリア + 好み/NG）
- 集計/提案画面（候補日程の強さ + 集合場所 + 店 2〜3 件 + AI 提案）

## 8. データ取り扱い・プライバシー（方針）

ログイン不要のため、ユーザーが安心して使える最小限のデータ設計を行います。

- 位置情報は、必要以上の精度を求めない（例: 住所そのものではなくエリア/駅レベルを基本）
- 取得方法はユーザーに明示し、許可しない場合は手入力に切り替え可能
- 共有 URL を知っている人が閲覧できる前提のため、個人が特定される情報の表示は抑制する

## 9. 非機能要件（最低限）

- モバイルでも操作しやすいカレンダー入力
- 入力の手戻りが少ない（後から編集可能）
- レスポンスが遅い場合でも状態が分かる（読み込み表示など）

## 10. スコープ外（本書の前提）

以下は現時点では必須要件に含めません。

- ユーザーアカウント/ログイン機能
- 高度な権限管理（閲覧制限の細分化など）
- 予約代行/決済
- 詳細な嗜好学習（履歴に基づく長期最適化）

## 11. 用語

- 調整: 日程・場所・店提案をまとめた単位
- 幹事: 調整を作成し確定するユーザー
- 参加者: URL から回答するユーザー
