# ミート&イート プロダクト概要書

最終更新: 2025-12-30

## 1. プロダクト概要

「ミート&イート」は、日程調整（調整さんのような体験）から集合場所の最適化、近辺の飲食店提案までを、**ログイン不要**で一気通貫に行える日程調整プロダクトです。

ユーザーはカレンダー型 UI で直感的に参加可能日を入力できます。入力時に、居住エリア（最寄り駅）を**検索・入力**し、その情報をもとにシステムが**最適な集合場所**を導出します。さらに、集合場所周辺の飲食店を**検索・レコメンド**し、最適なお店を候補として提案します。

## 2. 解決したい課題

- 日程が決まっても「どこで会うか」で再調整が発生しがち
- 参加者の居住地が分散していると集合場所の合意形成が難しい
- お店選びは好み・予算・アクセスなど制約が多く、提案が属人化しやすい

## 3. 想定ユーザー

- 友人同士の食事会/飲み会の幹事・参加者
- 複数人で集まるコミュニティ/サークル
- 社内の数懇親（※ログイン不要運用を前提）

## 4. 価値提案（提供する体験）

- **日程入力が早い**: カレンダー型 UI で直感的に可否を入力
- **場所が決まる**: 参加者の位置情報（最寄り駅）から集合場所（最適駅）を自動導出
- **店が決まる**: 周辺から飲食店を検索・提案
- **始めやすい**: ログイン不要で、URL 共有だけで参加できる

## 5. 主要なユーザーフロー

### 5.1 幹事（作成者）

1. 調整を作成（タイトル/候補日程など）
2. 共有 URL を発行し、参加者へ送付
3. 回答が集まる
4. システムが集合場所を導出し、飲食店候補を提示
5. 幹事が候補から選択し、イベントを確定

### 5.2 参加者

1. 共有 URL を開く（ログイン不要）
2. 居住エリア（最寄り駅）を検索・入力
3. （スキップ）食の好み/NG入力は現在未実装
4. カレンダーで参加可能日程を入力
5. 送信

## 6. 機能要件（MVP）

### 6.1 日程調整

- カレンダー型 UI で可否入力
- 回答の集計表示（誰がどこに入れたか/どの日が強いか）

### 6.2 居住エリア情報の取得

- 「最寄り駅」の検索・入力による取得（Google Maps / Nominatim APIを利用）
- 自身の出発地点（位置情報）を正確かつ直感的に入力可能

### 6.3 集合場所の最適化

- 参加者全員の地理的中心（Centroid）を計算
- 中心点周辺の駅を候補として抽出
- 各参加者からの移動時間（Transit Matrix）を計算・考慮し、全員にとって最も負担の少ない「最適駅」を導出
- 出力は「集まりやすい駅」として提示

### 6.4 飲食店レコメンド（2〜3 件）

- 決定した集合場所周辺の飲食店を HotPepper グルメ API から検索して提示
- ジャンルや予算などを表示し、リンクから詳細を確認可能
- （AI提案機能は現在フェーズではAPIによる検索結果の提示を主とする）

### 6.5 ログイン不要

- アカウント作成・ログインなしで作成/回答が可能
- 共有 URL で同一調整にアクセス

## 7. 画面/情報設計（概要）

- LP（説明・開始導線）
- 調整作成画面
- 回答画面（カレンダー入力 + 居住エリア + 好み/NG）
- 集計/提案画面（候補日程の強さ + 集合場所 + 店 2〜3 件 + AI 提案）

## 8. データ取り扱い・プライバシー（方針）

ログイン不要のため、ユーザーが安心して使える最小限のデータ設計を行います。

- 位置情報は、必要以上の精度を求めない（例: 住所そのものではなくエリア/駅レベルを基本）
- 取得方法はユーザーに明示し、許可しない場合は手入力に切り替え可能
- 共有 URL を知っている人が閲覧できる前提のため、個人が特定される情報の表示は抑制する

## 9. 非機能要件（最低限）

- モバイルでも操作しやすいカレンダー入力
- 入力の手戻りが少ない（後から編集可能）
- レスポンスが遅い場合でも状態が分かる（読み込み表示など）

## 10. スコープ外（本書の前提）

以下は現時点では必須要件に含めません。

- ユーザーアカウント/ログイン機能
- 高度な権限管理（閲覧制限の細分化など）
- 予約代行/決済
- 詳細な嗜好学習（履歴に基づく長期最適化）

## 11. 用語

- 調整: 日程・場所・店提案をまとめた単位
- 幹事: 調整を作成し確定するユーザー
- 参加者: URL から回答するユーザー
